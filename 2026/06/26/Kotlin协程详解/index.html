Kotlin协程详解

2026/6/26 Android Android Kotlin

协程是 Kotlin 官方推荐的异步编程方案，它用同步的写法完成异步操作，解决了传统回调地狱和线程切换的痛点。无论是网络请求、数据库操作还是并发任务，协程都能让代码简洁且高效。

什么是协程？

协程（Coroutine）是一种轻量级线程，它可以在不阻塞线程的情况下挂起和恢复执行。核心特点：

  • 挂起不阻塞：协程被挂起时，当前线程不会被阻塞，可以继续执行其他任务
  • 同步写法异步执行：用看似同步的代码写异步逻辑，告别回调嵌套
  • 结构化并发：协程有明确的生命周期作用域，任务可以被统一管理、取消

协程 vs 线程

特性 线程 协程
调度方式 操作系统调度 用户态调度
创建开销 较大（~1MB 栈空间） 极小（几 KB）
切换开销 内核态切换，成本高 用户态切换，成本低
数量限制 受内核限制 可创建成千上万个
取消支持 需要手动中断 内置取消机制

为什么需要协程？

1. 解决回调地狱

传统回调嵌套层层深入，可读性极差：

// 回调地狱
api.login(user) { token ->
    api.getProfile(token) { profile ->
        api.getFriends(profile.id) { friends ->
            // 终于拿到了 friends...
        }
    }
}

// 协程写法：线性逻辑，一目了然
suspend fun loadFriends() {
    val token = api.login(user)
    val profile = api.getProfile(token)
    val friends = api.getFriends(profile.id)
}

2. 线程切换简洁

传统方式需要 Handler、RxJava 等来回切换线程，协程用 withContext 一行搞定：

// 传统写法
Thread {
    val data = fetchData()
    runOnUiThread {
        textView.text = data
    }
}.start()

// 协程写法
lifecycleScope.launch {
    val data = withContext(Dispatchers.IO) { fetchData() }
    textView.text = data  // 自动切回主线程
}

3. 生命周期感知

协程配合 lifecycleScope 或 viewModelScope，当页面销毁时自动取消，避免内存泄漏和空指针。

协程基础

1. 创建协程：launch 与 async

// launch：不关心返回值，返回 Job
val job = scope.launch {
    doSomething()
}

// async：需要返回值，返回 Deferred
val deferred = scope.async {
    computeResult()
}
val result = deferred.await()  // 等待并获取结果

核心区别：launch 是”发射后不管”，async 是”执行并等待结果”。

2. 挂起函数 suspend

suspend 关键字标记的函数可以被挂起和恢复，只能在协程或其他挂起函数中调用：

suspend fun fetchUser(): User {
    delay(1000)  // 挂起，不阻塞线程
    return User("张三")
}

// delay 是非阻塞的挂起函数
// Thread.sleep 是阻塞的，两者完全不同

3. runBlocking

runBlocking 会阻塞当前线程直到协程完成，一般只用于测试和 main 函数：

fun main() = runBlocking {
    launch {
        delay(1000)
        println("Hello")
    }
    println("World")  // 先输出 World，1 秒后输出 Hello
}

协程调度器 Dispatchers

调度器决定了协程在哪个线程或线程池上执行：

lifecycleScope.launch(Dispatchers.Main) {
    // Dispatchers.Main：主线程，用于 UI 更新
    textView.text = "开始加载"

    val data = withContext(Dispatchers.IO) {
        // Dispatchers.IO：IO 线程池，用于网络、数据库等
        api.fetchData()
    }

    val processed = withContext(Dispatchers.Default) {
        // Dispatchers.Default：CPU 密集型线程池
        processLargeData(data)
    }

    textView.text = processed  // 回到主线程
}
调度器 用途 线程数
Dispatchers.Main UI 操作 主线程
Dispatchers.IO 网络、文件、数据库 64 个（可弹性扩展）
Dispatchers.Default CPU 密集型计算 CPU 核心数
Dispatchers.Unconfined 不限制（几乎不用） -

协程上下文与作用域

协程上下文 CoroutineContext

上下文是协程的配置集合，包含 Job、Dispatcher、CoroutineName 等元素：

scope.launch(
    Dispatchers.IO + Job() + CoroutineName("MyCoroutine")
) {
    // 协程名 + IO 调度器 + 独立 Job
}

// 通过 coroutineContext 获取
launch {
    println(coroutineContext[CoroutineName])  // 获取协程名
    println(coroutineContext[Job])            // 获取 Job
}

协程作用域 CoroutineScope

作用域管理协程的生命周期，不同作用域适用于不同场景：

// 1. GlobalScope：全局作用域，生命周期与 App 一致（不推荐）
GlobalScope.launch { }

// 2. lifecycleScope：Activity/Fragment 生命周期作用域
lifecycleScope.launch { }

// 3. viewModelScope：ViewModel 生命周期作用域
viewModelScope.launch { }

// 4. 自定义作用域
private val scope = CoroutineScope(Dispatchers.IO + SupervisorJob())
// 不再需要时取消
scope.cancel()

优先级推荐：viewModelScope > lifecycleScope > 自定义 Scope > GlobalScope

结构化并发

结构化并发是协程的核心设计理念：父协程必须等待所有子协程完成，形成了天然的依赖关系。

父子关系与取消传播

scope.launch {  // 父协程
    launch {  // 子协程 1
        delay(1000)
        println("子协程 1 完成")
    }
    launch {  // 子协程 2
        delay(2000)
        println("子协程 2 完成")
    }
    println("父协程完成")
}
// 父协程会等待所有子协程结束后才结束
// 取消父协程 = 取消所有子协程

supervisorScope vs coroutineScope

// coroutineScope：一个子协程失败 → 所有子协程取消
coroutineScope {
    launch {
        delay(500)
        throw RuntimeException("失败了！")
    }
    launch {
        delay(1000)  // 这个会被取消
    }
}

// supervisorScope：一个子协程失败 → 不影响其他子协程
supervisorScope {
    launch {
        delay(500)
        throw RuntimeException("失败了！")
    }
    launch {
        delay(1000)  // 这个正常执行
        println("我活下来了")
    }
}

协程的取消与会话

取消协程

val job = scope.launch {
    repeat(1000) { i ->
        delay(100)
        println("执行第 $i 次")
    }
}

delay(1500)
job.cancel()      // 取消协程
job.join()        // 等待协程结束
// 或一步到位
job.cancelAndJoin()

让协程可取消

协程的取消是协作式的，需要代码配合检查：

// 方式一：调用挂起函数（delay、withContext 等会自动检查取消）
launch {
    delay(1000)  // 取消时会抛 CancellationException
}

// 方式二：手动检查 isActive
launch {
    repeat(1000) { i ->
        if (!isActive) return@launch  // 主动检查
        // 耗时计算...
    }
}

// 方式三：ensureActive()
launch {
    repeat(1000) { i ->
        ensureActive()  // 已取消则抛 CancellationException
        // 耗时计算...
    }
}

// 方式四：yield() 让出执行权
launch {
    repeat(1000) { i ->
        yield()  // 挂起并检查取消
        // 耗时操作...
    }
}

资源释放：try-finally 和 use

launch {
    try {
        doLongRunningWork()
    } finally {
        // 清理资源（如关闭数据库连接）
    }
}

// 更推荐 use（类似 Java 的 try-with-resources）
launch {
    openResource().use { resource ->
        processResource(resource)
    }
}

并行执行多个任务

async + awaitAll

suspend fun loadAllData(): Triple<User, List<Post>, Config> = coroutineScope {
    // 三个请求同时发起，提速 3 倍
    val userDeferred = async { api.getUser() }
    val postsDeferred = async { api.getPosts() }
    val configDeferred = async { api.getConfig() }

    // 统一等待结果
    Triple(
        userDeferred.await(),
        postsDeferred.await(),
        configDeferred.await()
    )
}

// 更简洁的写法
suspend fun loadAllData(): Triple<User, List<Post>, Config> = coroutineScope {
    val (user, posts, config) = awaitAll(
        async { api.getUser() },
        async { api.getPosts() },
        async { api.getConfig() }
    )
    Triple(user, posts, config)
}

并发限制

// 使用 Semaphore 限制并发数
val semaphore = Semaphore(3) // 最多 3 个并发

val urls = listOf(...)
urls.map { url ->
    async {
        semaphore.withPermit {
            downloadFile(url)
        }
    }
}.awaitAll()

协程异常处理

try-catch + launch

// launch 内用 try-catch
lifecycleScope.launch {
    try {
        api.fetchData()
    } catch (e: IOException) {
        // 处理网络异常
    }
}

async 的异常

async 内部的异常只在调用 await() 时抛出：

val deferred = scope.async {
    throw RuntimeException("出错了！")
}

try {
    deferred.await() // 异常在这里抛出
} catch (e: Exception) {
    // 处理异常
}

CoroutineExceptionHandler

顶层的全局异常处理器，只能用于 launch（根协程），async 不适用：

val handler = CoroutineExceptionHandler { _, exception ->
    Log.e("Coroutine", "捕获异常：${exception.message}")
}

scope.launch(handler) {
    throw RuntimeException("测试异常")
}

异常传播规则

协程类型 异常行为
launch 异常直接抛出，父协程可能被取消
async 异常在 await() 时抛出
supervisorScope 子协程异常不影响兄弟协程
coroutineScope 一个失败全取消

实际应用场景

场景一：网络请求 + UI 更新

class UserViewModel : ViewModel() {
    private val _user = MutableLiveData<User>()
    val user: LiveData<User> = _user

    fun loadUser(userId: String) {
        viewModelScope.launch {
            // 在 IO 线程请求
            val user = withContext(Dispatchers.IO) {
                api.getUser(userId)
            }
            // 自动回到主线程
            _user.value = user
        }
    }
}

场景二：超时控制

lifecycleScope.launch {
    val result = withTimeout(3000) {  // 3 秒超时
        api.fetchData()
    }
    // 超时会抛 TimeoutCancellationException

    // 或者带默认值
    val result2 = withTimeoutOrNull(3000) {
        api.fetchData()
    } ?: "默认数据"
}

场景三：并发网络请求

lifecycleScope.launch {
    val (user, posts) = awaitAll(
        async(Dispatchers.IO) { api.getUser() },
        async(Dispatchers.IO) { api.getPosts() }
    )
    updateUI(user, posts)
}

场景四：Flow 流式数据

Flow 是协程中的响应式流，适合处理连续的数据序列：

// 创建 Flow
fun getMessages(): Flow<Message> = flow {
    while (true) {
        val msg = api.pollMessage()
        emit(msg)       // 发射数据
        delay(500)
    }
}

// 收集 Flow
lifecycleScope.launch {
    getMessages().collect { message ->
        updateMessageList(message)
    }
}

Flow 的冷流特性意味着数据只在被收集时才产生，不会浪费资源。此外，Flow 支持丰富的操作符：

lifecycleScope.launch {
    getMessages()
        .filter { it.type == MessageType.TEXT }  // 过滤
        .map { it.content.uppercase() }           // 转换
        .flowOn(Dispatchers.IO)                   // 切换上游线程
        .catch { e -> emit("加载失败") }           // 异常处理
        .collect { text -> updateUI(text) }
}

场景五：防抖处理

// 搜索框输入防抖
class SearchViewModel : ViewModel() {
    private var searchJob: Job? = null

    fun onSearchTextChanged(query: String) {
        searchJob?.cancel()  // 取消上次搜索
        searchJob = viewModelScope.launch {
            delay(300)  // 300ms 内无新输入才真正搜索
            val results = withContext(Dispatchers.IO) {
                api.search(query)
            }
            _searchResults.value = results
        }
    }
}

场景六：重试机制

suspend fun <T> retry(
    times: Int = 3,
    initialDelay: Long = 1000,
    block: suspend () -> T
): T {
    var currentDelay = initialDelay
    repeat(times - 1) {
        try {
            return block()
        } catch (e: Exception) {
            delay(currentDelay)
            currentDelay *= 2  // 指数退避
        }
    }
    return block()  // 最后一次尝试
}

// 使用
val data = retry(times = 3) {
    api.fetchUnstableData()
}

常见问题及解决方案

1. NetworkOnMainThreadException

协程不会自动切换线程！在 Dispatchers.Main 启动的协程做网络请求仍然会崩溃。

错误写法：

lifecycleScope.launch {
    api.fetchData()  // 崩溃！仍在主线程
}

正确写法：

lifecycleScope.launch {
    val data = withContext(Dispatchers.IO) {
        api.fetchData()  // 正确：在 IO 线程
    }
    updateUI(data)
}

2. 协程泄露

启动协程但不管理生命周期，导致协程在页面销毁后仍在执行。

// 错误：GlobalScope 不受生命周期管控
GlobalScope.launch {
    api.fetchData()  // 页面销毁后仍在执行
}

// 正确：使用生命周期作用域
lifecycleScope.launch {
    api.fetchData()  // 页面销毁自动取消
}

3. 取消后执行不可取消代码

被取消的协程中，非挂起代码可能继续执行：

val job = scope.launch {
    try {
        while (hasMore) {
            processItem()  // 纯计算，不会检查取消
        }
    } finally {
        // 这里仍然会执行
        cleanUp()
    }
}
job.cancel()

解决方案：在循环中主动检查 isActive。

4. suspend 函数只能在协程内调用

// 编译错误：Suspend function should be called only from a coroutine
fun onClick() {
    fetchUser()  // 错误！
}

// 正确
fun onClick() {
    lifecycleScope.launch {
        fetchUser()  // 正确
    }
}

有哪些最佳实践？

1. 用 viewModelScope / lifecycleScope，不用 GlobalScope

class MyViewModel : ViewModel() {
    fun loadData() {
        viewModelScope.launch {  // 推荐
            // 页面销毁自动取消
        }
    }
}

2. withContext 切换线程，不要让 launch 的调度器混用

// 推荐：外层用 Main，内部用 withContext 切换
lifecycleScope.launch(Dispatchers.Main) {
    val data = withContext(Dispatchers.IO) { api.fetch() }
    updateUI(data)
}

// 不推荐：外层直接指定 IO，回主线程又得套一层
lifecycleScope.launch(Dispatchers.IO) {
    val data = api.fetch()
    withContext(Dispatchers.Main) { updateUI(data) }
}

3. 避免不必要的 async

如果不需要并发，不要用 async → await，直接用 withContext：

// 不推荐：单任务用 async
val result = scope.async(Dispatchers.IO) { api.fetch() }.await()

// 推荐：单任务用 withContext
val result = withContext(Dispatchers.IO) { api.fetch() }

4. 用 SupervisorJob 隔离异常

// ViewModel 中推荐 SupervisorJob
private val scope = CoroutineScope(SupervisorJob() + Dispatchers.Main)
scope.launch { task1() }  // task1 失败不影响 task2
scope.launch { task2() }

5. Flow 中使用 catch 而非 try-catch

// 推荐
flow {
    emit(api.fetch())
}.catch { e ->
    emit("默认值")  // Flow 内建的异常处理
}.collect { ... }

// 不推荐（异常会取消 Flow）
flow.collect {
    try {
        // ...
    } catch (e: Exception) { ... }
}

6. 及时释放资源

launch {
    Dispatchers.IO.use {  // use 会自动清理
        val channel = openChannel()
        // ...
    }
}

总结

协程是 Kotlin 异步编程的首选方案，核心优势：

  • 同步写法，异步执行：suspend 函数让异步代码读起来像同步代码
  • 轻量级：协程不是线程，创建和切换成本极低，可以开成千上万个
  • 结构化并发：coroutineScope / supervisorScope 管理父子关系，取消传播自动化
  • 线程切换简单：withContext 一行搞定，不需要 Handler、RxJava 等工具

记住核心口诀：launch 不管结果、async 等结果、suspend 可挂起、withContext 切线程。

最后牢记三点：

  1. 用 viewModelScope / lifecycleScope 绑定生命周期，别用 GlobalScope
  2. 耗时操作用 withContext(Dispatchers.IO) 包裹，别让主线程协程执行耗时任务
  3. 循环耗时计算中用 isActive 检查取消状态，保证协程可取消
