Activity的四种启动方式

2026/3/15 Android Android

Activity 是 Android 四大组件中最常用的组件，它的启动模式决定了 Activity 实例如何与任务栈交互。理解这四种启动模式是掌握 Android 导航和任务管理的关键。

什么是 Activity 的启动模式？

Activity 的启动模式（Launch Mode）定义了当启动一个 Activity 时，该 Activity 实例如何与当前任务栈（Task Stack）关联。Android 提供了四种启动模式，每种都有不同的实例化和复用策略。

为什么需要不同的启动模式？

  • 避免重复创建：防止同一个 Activity 被反复创建，浪费内存
  • 控制返回逻辑：按返回键时能回到用户期望的页面
  • 优化用户体验：让页面跳转逻辑更符合用户直觉

四种启动模式详解

1. standard（标准模式）

系统默认的启动模式，每次启动 Activity 都会创建一个新的实例。

特点：

  • 每次 startActivity() 都会创建新实例
  • 新实例压入启动它的任务栈顶部
  • 同一个 Activity 可以在栈中存在多个实例

配置方式：

<!-- AndroidManifest.xml -->
<activity
    android:name=".StandardActivity"
    android:launchMode="standard" />

示意图：

假设当前栈为 [A, B]，从 B 启动 standard 模式的 C 两次：

启动第一次C： [A, B, C1]
启动第二次C： [A, B, C1, C2]

按返回键会依次回到 C1 → B → A。

适用场景：绝大多数普通页面，如详情页、列表页等。

2. singleTop（栈顶复用模式）

如果目标 Activity 已经在当前任务栈的栈顶，则不会创建新实例，而是调用该实例的 onNewIntent() 方法；如果不在栈顶，则创建新实例。

特点：

  • 栈顶已存在 → 复用，触发 onNewIntent()
  • 不在栈顶 → 创建新实例
  • 可以存在多个实例（只要不都在栈顶）

配置方式：

<activity
    android:name=".SingleTopActivity"
    android:launchMode="singleTop" />

示意图：

情况一：栈为 [A, B, C]，从 C 启动 singleTop 模式的 C：

启动C（已在栈顶）： [A, B, C]  ← 复用，触发onNewIntent()

情况二：栈为 [A, B]，从 B 启动 singleTop 模式的 C：

启动C（不在栈顶）： [A, B, C]  ← 创建新实例

// 在 singleTop 的 Activity 中接收新的 Intent
@Override
protected void onNewIntent(Intent intent) {
    super.onNewIntent(intent);
    // 处理新的 Intent 数据
    setIntent(intent);
    String newData = intent.getStringExtra("key");
    // 更新页面数据
}

适用场景：

  • 消息推送点击通知打开的页面
  • 搜索页面（避免重复打开）
  • WebView 页面

3. singleTask（栈内复用模式）

如果目标 Activity 已经在当前任务栈中存在，则将该 Activity 上面的所有 Activity 全部出栈，使其回到栈顶，并调用 onNewIntent()。如果不存在，则创建新实例。

特点：

  • 全局只有一个实例（在当前任务栈中）
  • 会把上方所有 Activity 清除出栈（clearTop 效果）
  • 默认在新的任务栈中不一定创建新栈

配置方式：

<activity
    android:name=".SingleTaskActivity"
    android:launchMode="singleTask" />

示意图：

栈为 [A, B, C, D]，从 D 启动 singleTask 模式的 B：

启动前： [A, B, C, D]
启动后： [A, B]  ← C和D被清除，B收到onNewIntent()

适用场景：

  • 应用的主页/首页
  • 浏览器的主页
  • 需要作为入口点的页面

4. singleInstance（单实例模式）

最特殊的一种模式，该 Activity 会在一个全新的独立任务栈中创建，并且这个新任务栈中只能有这一个 Activity。全局唯一。

特点：

  • 独占一个任务栈
  • 全局只有一个实例
  • 如果已存在，直接切换到该任务栈并调用 onNewIntent()

配置方式：

<activity
    android:name=".SingleInstanceActivity"
    android:launchMode="singleInstance" />

示意图：

Task1: [A, B, C]
Task2: [X]  ← singleInstance模式，独占一个任务栈

从 C 启动 X：系统切换到 Task2，显示 X。
从 X 按返回键：回到 Task1 的 C。

适用场景：

  • 来电铃声页面
  • 系统闹钟页面
  • 与其他应用完全隔离的页面
  • 视频全屏播放页面

四种模式对比总结

启动模式 是否新建实例 是否复用 清除栈上方 独立任务栈
standard 总是新建 否 否 否
singleTop 栈顶时复用 仅栈顶 否 否
singleTask 存在时复用 栈内任一 是 否
singleInstance 存在时复用 全局唯一 不适用 是

常见问题有哪些？

1. onNewIntent 什么时候被调用？

问题：设置了 singleTop 或 singleTask 后，数据没有更新？

解决方案：onNewIntent() 只在 Activity 被复用时调用。第一次创建时走 onCreate() 的正常流程。需要同时处理两种情况：

@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    setContentView(R.layout.activity_main);
    handleIntent(getIntent());
}

@Override
protected void onNewIntent(Intent intent) {
    super.onNewIntent(intent);
    setIntent(intent);
    handleIntent(intent);
}

private void handleIntent(Intent intent) {
    String data = intent.getStringExtra("key");
    // 统一处理数据更新
}

2. singleTask 和 singleInstance 的区别

问题：什么时候用 singleTask，什么时候用 singleInstance？

singleTask 和 singleInstance 的核心区别在于任务栈的隔离：

  • singleTask：在当前任务栈中保持唯一，但和同栈的其他 Activity 共享返回栈
  • singleInstance：完全独立的任务栈，和其他 Activity 隔离
// singleTask：回到主页面，清除登录页和注册页
// 场景：登录成功后跳转到主页
Intent intent = new Intent(this, MainActivity.class);
intent.setFlags(Intent.FLAG_ACTIVITY_CLEAR_TOP);
startActivity(intent);

// singleInstance：完全隔离的页面
// 场景：视频播放页面，不希望被其他页面干扰

3. Intent Flag 和 launchMode 的优先级

问题：如果同时在代码里设置了 Flag，又在 Manifest 里配置了 launchMode，哪个生效？

解决方案：代码中通过 Intent.setFlags() 设置的优先级更高，会覆盖 Manifest 中的配置。

常用 Flag：

Flag 效果
FLAG_ACTIVITY_NEW_TASK 在新任务栈中启动
FLAG_ACTIVITY_SINGLE_TOP 相当于 singleTop
FLAG_ACTIVITY_CLEAR_TOP 清除上方的 Activity
FLAG_ACTIVITY_CLEAR_TASK 启动前清空任务栈 
// 代码中覆盖为 singleTop 行为
Intent intent = new Intent(this, DetailActivity.class);
intent.addFlags(Intent.FLAG_ACTIVITY_SINGLE_TOP);
startActivity(intent);

4. 如何验证启动模式的效果？

可以通过 adb 命令查看当前任务栈的情况：

# 查看任务栈信息
adb shell dumpsys activity activities | grep -A 5 "mResumedActivity\|mFocusedActivity\|Activities="

或者在 Activity 的生命周期回调中打印日志来跟踪：

@Override
protected void onCreate(Bundle savedInstanceState) {
    super.onCreate(savedInstanceState);
    Log.d("LaunchMode", "onCreate: " + this.getClass().getSimpleName() 
          + ", TaskId: " + getTaskId() 
          + ", HashCode: " + hashCode());
}

有哪些最佳实践？

1. 合理选择启动模式

  • 普通页面：使用默认的 standard
  • 通知点击入口：使用 singleTop，避免重复打开
  • 应用主页：使用 singleTask，确保返回时不会回退到登录页
  • 独立功能页：使用 singleInstance，如通话、闹钟

2. 设计清晰的任务栈结构

合理结构：
[主页(singleTask)] → [列表页(standard)] → [详情页(standard)]

错误结构：
[主页] → [主页] → [主页]  ← 多个主页实例，返回体验差

3. 处理 onNewIntent 数据刷新

确保同时重写 onCreate 和 onNewIntent，避免数据不更新的问题。

如何验证和测试？

一个简单的测试方案：在每个 Activity 中打印日志和 Toast。

public class BaseActivity extends AppCompatActivity {
    @Override
    protected void onCreate(Bundle savedInstanceState) {
        super.onCreate(savedInstanceState);
        String info = this.getClass().getSimpleName() 
            + " | taskId=" + getTaskId() 
            + " | instance=" + hashCode();
        Log.d("ActivityTest", "onCreate: " + info);
        Toast.makeText(this, "Created: " + info, Toast.LENGTH_SHORT).show();
    }

    @Override
    protected void onNewIntent(Intent intent) {
        super.onNewIntent(intent);
        String info = this.getClass().getSimpleName() + " | onNewIntent";
        Log.d("ActivityTest", info);
        Toast.makeText(this, info, Toast.LENGTH_SHORT).show();
    }
}

然后让所有 Activity 继承 BaseActivity，通过点击按钮触发跳转，观察日志和 Toast 输出来理解每种模式的行为。

总结

Activity 的四种启动模式各有特点，理解它们的关键在于任务栈这个概念：

  • standard：来者不拒，每次都新建
  • singleTop：看栈顶，在顶上就复用
  • singleTask：查全栈，找到就把上面的清掉
  • singleInstance：搞特殊，独享一个栈

选择合适的启动模式能让你的应用在返回逻辑、内存管理、用户体验上都更加优秀。建议实际动手写个 Demo，多点点看看效果，配合 adb shell dumpsys activity 观察任务栈的变化，理解会更加深刻。
